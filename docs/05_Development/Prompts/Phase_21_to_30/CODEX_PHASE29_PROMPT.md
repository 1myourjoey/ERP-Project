# Phase 29: 전사 시스템 차원의 정보 구조(IA) 및 UX 효율화 마스터 플랜

> **Priority:** P1 (System-wide Information Architecture Refactoring)

---

## 🚨 치명적 핵심 원칙 (Non-negotiable Rules): "기능적 무결성 100% 보존"

Phase 28과 마찬가지로, 본 페이즈는 오직 시스템의 **"정보 구조(IA) 라우팅과 UX/UI 레이아웃"**만을 글로벌 SaaS 표준으로 격상시키는 작업입니다.
**어떠한 경우에도 기존에 완벽하게 동작하고 있던 백엔드 API 연동, 데이터 저장/수정 로직, 컴포넌트 간 상태 값(State)이 변경되거나 유실되어서는 안 됩니다.**

1. **Zero Backend Modification:** `InvestmentsPage`, `WorkflowsPage` 내부에서 호출하던 React Query 및 Mutate 함수 등 데이터 I/O 기능은 단 1mg도 변형하지 마십시오. 오직 화면에 그리는 위치(View)와 서랍(Drawer) 방식으로만 전환하십시오.
2. **연쇄 Invalidation 생존 보장:** 회사를 추가하거나 워크플로 템플릿을 수정했을 때 연쇄적으로 발동하던 캐시 무효화(`queryClient.invalidateQueries`) 로직이 모달 -> 슬라이드 오버(Drawer)로 바뀌는 과정에서 누락되지 않도록 Props 연결에 극도로 유의하십시오.
3. **Draft State(입력 상태) 방어:** 우측 슬라이드 서랍창을 열고 타이핑을 하던 중, 좌측 엑셀 뷰를 스크롤한다고 해서 폼 입력 상태가 증발하는 일이 절대 없도록 상태 관리를 철저히 격리하십시오.

---

## 🚨 현황 진단 및 핵심 과제 (Why we need this)
Phase 28과 28_1을 통해 **"조합(Fund)"** 도메인에 있어서는 [전역 대시보드]와 [개별 상세 통제실]이라는 완벽한 분업화 및 모던 탭(Tab) UX 구조를 확립했습니다. 
Namun(하지만), V:ON ERP의 다른 핵심 도메인 체계들(**투자(Investments), 워크플로(Workflows), 타스크(Tasks)**)은 여전히 초기 구조의 한계인 **"단일 페이지 무한 레이어 확장"** 패턴에 갇혀 실무 효율을 떨어뜨리고 있습니다.

이번 Phase 29에서는 전체 ERP 시스템의 UX 사상(Mental Model)을 일관되게 맞추고, 각 도메인별 페인포인트(Pain-point)를 정밀 타격하여 SaaS로서의 완성도를 극한으로 끌어올립니다.

## Part 1. `InvestmentsPage` (투자 관리) -> [목록]과 [상세]의 분리

**[문제점]**
현재 `InvestmentsPage.tsx`는 화면 왼쪽 절반에 '포트폴리오 회사 목록(+ 수정 폼)', 오른쪽 절반에 '투자 목록(+ 거대한 세로형 투자 등록 폼)'을 동시에 우겨넣고 있는 **대시보드형 콤보 구조**입니다. 항목이 많아질수록 화면 높이가 미친 듯이 늘어나고, 회사별 누적 투자 히스토리나 첨부파일 관리가 불가능합니다.

**[해결책: IA 재설계]**
1. **`InvestmentsPage.tsx` (전역 투자 대시보드):** 
   - 좌우 분할 레이아웃을 폐기합니다.
   - 단일 화면 전체를 거대한 **[투자 포트폴리오 엑셀 그리드]**로 전환합니다. (펀드명, 회사명, 투자수단, 총액, 현재 밸류, 수익률 등을 한눈에 모니터링)
   - '회사관리' 기능은 상단 네비게이션 드롭다운이나 별도 탭으로 분리하여 목록만 심플하게 보여줍니다.
2. **`InvestmentDetailPage.tsx` (개별 투자 건 상세 통제실):** 
   - 펀드 상세뷰(`FundDetailPage`)처럼 개별 투자 건에 대해서도 단독 상세화면(`URL: /investments/:id`)으로 100% 진입하도록 유도합니다.
   - ಈ 페이지 내부에 5-Tab을 신설합니다: `① 투자 개요`, `② 사후 관리(밸류에이션, 권리)`, `③ 관련된 워크플로`, `④ 문서 및 계약서`, `⑤ 회수(Exit) 시뮬레이션`.

## Part 2. `WorkflowsPage` (워크플로 관리) -> 템플릿과 실행뷰의 이원화

**[문제점]**
`WorkflowsPage.tsx` 역시 1500줄이 넘어가는 괴물 컴포넌트입니다. **[진행 중인 워크플로 목록]**과 **[워크플로 템플릿 마스터 제작 기능]**이 한 화면에 탭(기존: 좌측 네비 형태)으로 엮여 있어, 실무자와 시스템 관리자의 동선이 꼬입니다.

**[해결책: IA 재설계]**
1. **관리자용 `TemplateManagementPage` 신설:** 
   - 일반 직원은 거의 접근하지 않는 "워크플로 템플릿 뼈대 짜기, 사분면 지정, 기본 문서 할당" 기능 전체를 아예 별도의 시스템 설정 페이지(`메뉴 > 시스템/템플릿 관리`)로 뜯어냅니다.
2. **실무자용 `WorkflowsPage` 경량화 & 뷰(View) 강화:** 
   - 오로지 "지금 어떤 펀드/회사에 대해 무슨 워크플로가 진행 중이고, 누가 며칠 지연 중인가(Red 라이트)"를 추적하는 **칸반(Kanban) 보드**나 **간트(Gantt) 차트 뷰** 형태의 UI 컴포넌트를 이식합니다.

## Part 3. 공통 UX/UI 효율성 개선 (Global Polish)

**1. 모달(Modal) 대신 슬라이드 오버 (Slide-Overs) 도입**
   - 현재 LP 양도나 워크플로 템플릿 편집처럼 덩치가 큰 입력 양식들이 중앙 팝업 모달 형태로 떠서 화면 뒷배경 맥락을 완전히 차단합니다.
   - 거대한 폼 입력은 **우측에서 부드럽게 튀어나오는 Drawer (Slide-Over)** 로 교체하여, 사용자가 뒤에 깔린 엑셀 데이터 원본을 참고하면서 입력할 수 있게끔 UX를 향상시킵니다.

**2. 전역 스크롤(Window Scroll)에서 내부 컨테이너 스크롤(Inner Scroll)로 전환**
   - 상단 헤더(Header)와 사이드바(GNB)는 무조건 화면에 고정(Sticky)시키고, 오직 데이터 리스트 영역만 스크롤되도록 `<div className="overflow-y-auto h-[calc(100vh-...)]">` 구조를 모든 페이지에 일관되게 규격화합니다.

**3. 검색 및 필터 패널 컴포넌트 통일화 (`DataFilterBar.tsx`)**
   - 모든 페이지 상단에 흩어져 있는 Input 태그 기반 검색과 콤보박스들을 공통 컴포넌트인 `DataFilterBar` 로 묶어서, SaaS 제품 특유의 일관되고 깔끔한 상단 검색 바 영역을 구축합니다.

## Acceptance Criteria
- [ ] AC-01: UX 라우팅 개편 작업 중 기존 데이터 호출, 생성, 업데이트 로직이 단 1건도 파손되거나 유실되면 안 된다. (Regression Zero)
- [ ] AC-02: 전체 ERP에서 "목록을 보는 화면(대시보드)"과 "상세 업무를 하는 화면(통제실)"의 IA 라우팅 구조가 100% 일관성 있게 분리되어야 한다.
- [ ] AC-03: 거대하고 복잡한 입력/수정 폼(Form)은 기존 팝업 모달 대신 우측 Slide-Over 형태로 렌더링되어 Context 전환 비용을 최소화해야 한다.
- [ ] AC-04: `InvestmentsPage`와 `WorkflowsPage`의 코드 라인 수가 기존 대비 50% 수준으로 모듈화되어, 프론트엔드 유지보수성이 혁신적으로 확보되어야 한다.
