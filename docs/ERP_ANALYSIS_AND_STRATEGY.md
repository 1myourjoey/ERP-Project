## 1. 현재 시스템 현황

## 2. 아키텍처 분석

### 2.1 강점

| 영역 | 설명 |
|------|------|
| **도메인 모델링** | 68개 모델이 VC 업무 전반을 잘 커버함. Fund를 중심으로 LP, Investment, CapitalCall, Distribution, Exit이 유기적으로 연결 |
| **서비스 레이어 분리** | 비즈니스 로직이 `services/`에 분리되어 라우터가 얇음 (컴플라이언스 엔진, 워크플로우 서비스, 문서생성 등) |
| **워크플로우 엔진** | D-day 기반 일정 자동계산, 영업일 처리, 공휴일 반영, 주기적 스케줄링까지 실무에 최적화 |
| **컴플라이언스 엔진** | JSON 조건식 기반 규칙 평가, 자동 의무사항 생성, LLM 기반 법규 해석까지 고급 기능 |
| **React Query 활용** | 서버 상태와 클라이언트 상태 분리, `invalidateQueries`를 통한 캐시 무효화 패턴 일관성 |
| **인증 체계** | JWT + 리프레시 토큰 + Google OAuth + 역할 기반 접근제어 + 초대 기반 가입 |

### 2.2 구조적 위험

#### (1) SQLite 호환성 레이어의 복잡도 — **심각**

`main.py`가 **900줄 이상**의 SQLite 호환성 처리 코드를 포함. 테이블 컬럼 추가, 재구축, 기본값 채우기 등이 startup에서 매번 실행됨.

```
문제: 마이그레이션이 Alembic과 수동 코드 양쪽에서 관리되어 충돌 위험
영향: 스키마 변경 시 매번 양쪽 수정 필요, 운영 DB 손상 위험
권장: PostgreSQL 전환 또는 Alembic 단일 관리 체계로 정리
```

#### (2) api.ts 단일 파일 — **높음**

프론트엔드 API 레이어가 **4,451줄** 단일 파일. 400+ 함수가 한 파일에 집중.

```
문제: 유지보수 어려움, IDE 성능 저하, 코드 네비게이션 비효율
권장: 도메인별 분리 (tasksApi.ts, fundsApi.ts, complianceApi.ts 등)
```

#### (3) 레거시 이중 스키마 — **중간**

CapitalCall, Distribution에서 구형(Item)과 신형(Detail) 두 가지 포맷 공존.

```
문제: API 혼동, 코드 중복, 프론트엔드 분기 처리 증가
권장: 구형 포맷 deprecated 마킹 → v1.0에서 제거
```

## 3. 기능 연계성 분석

### 3.1 연계가 잘 되어 있는 부분

```
Fund ─── LP ─── CapitalCall ─── CapitalCallDetail (LP별)
  │              │
  │              └── LPContribution (출자 이력)
  │
  ├── Investment ─── Valuation ─── Transaction
  │      │
  │      └── ExitTrade ─── Distribution ─── DistributionDetail (LP별)
  │
  ├── WorkflowInstance ─── WorkflowStepInstance ─── Task
  │                              │
  │                              └── StepInstanceDocument (체크리스트)
  │
  └── ComplianceObligation ─── ComplianceCheck (감사 로그)
```

### 3.2 연계가 끊어진 부분 — **개선 필요**

#### A. 태스크 ↔ 회계 연결 부재
현재 태스크 완료가 회계 분개와 연결되지 않음. 예: 자본금 콜 워크플로우 완료 → 분개 자동생성이 불가.

```
현재: Task 완료 → WorkLog 기록 (수동)
개선: Task 완료 → 자동 분개 트리거 (source_type='workflow_step')
      예) 출자금 입금 확인 단계 완료 → 차) 보통예금 / 대) 출자금 분개 자동생성
```

#### B. 사업보고서 ↔ 밸류에이션 연결 부재
BizReport의 재무 데이터(매출, 영업이익)가 Valuation 모델과 연동되지 않음.

```
현재: BizReport의 financial metrics (NAV, IRR)와 Valuation이 별도 입력
개선: BizReport 승인 → Valuation 자동 업데이트 제안
      BizReport의 재무지표 → Valuation fair_value 재산정 트리거
```

#### C. 엑시트 ↔ 배분 ↔ 성과보수 자동 연결 부재
ExitTrade가 생성되어도 Distribution과 PerformanceFee가 자동으로 연결되지 않음.

```
현재: ExitTrade 생성 → 수동으로 Distribution 생성 → 수동으로 성과보수 계산
개선: ExitTrade 확정 → Distribution 자동생성 (LP별 pro-rata)
                     → PerformanceFee 자동 시뮬레이션
                     → 회계 분개 자동생성
```

#### D. 컴플라이언스 ↔ 문서생성 연결 미흡
컴플라이언스 의무사항이 특정 문서 제출을 요구할 때, 문서 템플릿과 연동이 안 됨.

```
현재: 의무사항 생성 → 수동으로 해당 문서 찾아서 생성
개선: ComplianceObligation에 template_id 필드 추가
      의무사항 뷰에서 "문서 생성" 버튼 → 해당 템플릿으로 바로 생성
```

#### E. 정기 스케줄 ↔ 대시보드 연결 미흡
PeriodicSchedule로 정의된 정기 업무가 대시보드에서 예고/알림 형태로 표시되지 않음.

```
현재: 주기적 스케줄은 백그라운드에서 태스크 생성
개선: 대시보드에 "이번 주 정기 업무" 섹션 추가
      D-3일 전부터 예고 카드 표시
      미생성 정기 업무에 대한 경고 표시
```

#### F. LP 주소록 ↔ LP(펀드) 데이터 미연동
LPAddressBook과 Fund의 LP 모델이 별도로 존재하지만 연동이 약함.

```
현재: LP 추가 시 주소록에서 별도 검색/입력
개선: LP 추가 시 주소록에서 자동완성
      주소록 정보 변경 시 연관 펀드 LP 정보 일괄 업데이트 제안
```

### 3.3 데이터 흐름 단절 지점

| 시작점 | 끝점 | 현재 상태 | 개선 방향 |
|--------|------|-----------|-----------|
| CapitalCall 완료 | LP.paid_in 업데이트 | 부분 자동화 | 완전 자동화 + 검증 |
| ExitTrade 확정 | Distribution 생성 | 수동 | 자동 pro-rata 배분 |
| Distribution 실행 | 회계 분개 | 수동 | 자동 분개 |
| BizReport 승인 | Valuation 반영 | 미연결 | 자동 반영 제안 |
| Compliance 위반 | Task 생성 | 부분 자동 | 완전 자동 + 에스컬레이션 |
| WorkflowStep 완료 | 다음 단계 알림 | 시스템 내부 | Slack/이메일 알림 |

## 4. 누락 기능 분석

### 4.1 핵심 누락 (1인 VC 관리자에게 즉시 필요)

| 우선순위 | 기능 | 이유 |
|----------|------|------|
| **P0** | **현금흐름 예측 (Cash Flow Projection)** | 펀드 운용의 핵심. 콜 일정, 수수료, 배분 예정을 통합한 현금흐름 뷰 부재 |

## 5. 코드 품질 개선점

### 5.1 Backend

| 등급 | 항목 | 설명 | 개선 방법 |
|------|------|------|-----------|
| **심각** | main.py 900줄 호환 코드 | SQLite 스키마 패치가 startup에 집중 | Alembic 단일 관리 또는 PostgreSQL 전환 |
| **높음** | N+1 쿼리 문제 | Task, Investment 상세 조회 시 관계 데이터 개별 조회 | `selectinload`/`joinedload` 적용 |
| **높음** | 영업일 계산 중복 | workflow_service, compliance_engine 등에 동일 로직 | `utils/business_days.py`로 통합 |
| **중간** | 소프트 삭제 미구현 | 모든 삭제가 하드 삭제 (CASCADE) | `is_deleted` + `deleted_at` 필드 추가 |
| **중간** | Rate Limiting 부재 | 문서생성, LLM 호출 등 비용 발생 API에 제한 없음 | FastAPI slowapi 미들웨어 |
| **중간** | 감사 로그 미활용 | AuditLog 모델 존재하나 대부분 미기록 | 미들웨어로 자동 로깅 |
| **낮음** | 환경변수 미정리 | 필요 환경변수가 코드에 흩어져 있음 | `.env.example` 파일 정비 |

### 5.2 Frontend

| 등급 | 항목 | 설명 | 개선 방법 |
|------|------|------|-----------|
| **높음** | api.ts 4,451줄 | 모든 API 함수가 단일 파일 | 도메인별 분리 (10~15개 파일) |
| **높음** | 폼 라이브러리 미사용 | useState로 수동 폼 관리 | react-hook-form + zod 도입 |
| **중간** | 코드 스플리팅 없음 | 35개 페이지 모두 초기 로드 | React.lazy + Suspense |
| **중간** | 에러 바운더리 없음 | 페이지 에러 시 전체 앱 크래시 | ErrorBoundary 컴포넌트 |
| **중간** | 상태값 하드코딩 | '대기', '진행중' 등 문자열 직접 사용 | enum/상수 파일로 통합 |
| **중간** | 쿼리 키 관리 | 문자열 배열로 직접 사용 | queryKeys 상수 객체 |
| **낮음** | 접근성 (a11y) 미흡 | ARIA 라벨, 포커스 관리 부족 | 점진적 개선 |
| **낮음** | 테스트 코드 부재 | 테스트 파일 0개 | 핵심 로직 단위 테스트부터 |

매주 금요일:

  □ 분기 LP 보고서 자동 초안 생성

  □ 사업보고서 수집 워크플로우 자동 시작

### 6.3 데이터 입력 효율화

#### 스마트 입력 제안

```
투자 입력 시:
  • 기업명 입력 → 기존 포트폴리오 기업 자동완성
  • 펀드 선택 → 잔여 투자 한도 실시간 표시
  • 금액 입력 → 투자 한도 초과 여부 즉시 경고
  • 라운드 선택 → 이전 라운드 투자 이력 표시

LP 추가 시:
  • LP명 입력 → 주소록에서 자동완성 (사업자번호, 주소, 연락처)
  • 약정금액 입력 → 펀드 목표 잔여액 표시
  • 유형 선택 → 해당 유형 LP의 규제 요건 안내
```
